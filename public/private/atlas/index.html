Chapter 1 · Lab 01 · Parallelism

Eight geniuses vs. four thousand interns

A CPU core is a genius: deep pipelines, branch prediction, huge caches — brilliant at one thing at a time. A GPU core is an intern: simple and slow. But a screen has millions of pixels and none of them care about each other, so you don't want a genius. You want a crowd.

sim clock: 0 cycles

CPU

8 cores · 5.7 GHz · out-of-order geniuses
0% of pixels shaded

GPU

4,096 cores · 2.2 GHz · slow, simple, everywhere at once
0% of pixels shaded
Each core shades one tile at a time. CPU cores are modeled 8× faster per core — and it doesn't matter. Timing is stylized; the ~20× gap is not.

This is the entire founding bet of the GPU: throughput over latency. A CPU minimizes the time to finish one task; a GPU maximizes tasks finished per second, and accepts that any individual task crawls. Pixel shading, physics, and — it turned out later — neural networks are all "embarrassingly parallel": millions of identical, independent little jobs.

Every design choice in the rest of this chapter falls out of that one bet.

Try: run the race, then imagine the workload were one long dependent calculation (each step needs the last). Which side wins then, and why?
Why can't we just build a 5.7 GHz chip with 4,096 genius cores?
Power and area. Out-of-order machinery (reorder buffers, branch predictors, big private caches) spends most of a CPU core's transistors and watts on scheduling work, not doing it. Strip all that out and each core shrinks ~50×, so thousands fit in the same silicon and power budget — as long as the work is regular enough to not need the fancy scheduling.

Anatomy of a die

GPU DIE · AD102-class · 608 mm²
Hover a component
—
Everything on this die is drawn roughly to its real proportions. The repeated tiles are the whole story: a GPU is one good idea, photocopied 144 times.

The unit that repeats is the Streaming Multiprocessor (SM) — NVIDIA's name; AMD calls theirs a Compute Unit. One SM is a complete little processor: schedulers, registers, math units, its own scratchpad memory. The die is mostly a grid of them wrapped around a huge shared L2 cache, with memory controllers around the edges talking to the VRAM chips that sit next to the die on the board.

Notice what's missing compared to a CPU die: almost no area is spent on control. It's math units and memory, wall to wall.

Try: click into an SM and find the four warp schedulers — then count how much of the SM is register file. Why would a processor need 256 KB of registers? Lab 04 answers this.

Chapter 1 · Lab 03 · Execution model

32 threads, one instruction: the warp

GPU threads don't run alone. They run in packs of 32 called warps, and every thread in the warp executes the same instruction at the same time on different data — SIMT, single instruction, multiple threads. It's why GPU cores can be so simple. It's also why if-statements are expensive.

cycles: 0 lane-cycles doing useful work: 0 utilization: —
■ copper = lane executing · ▨ hatched = lane masked off (burning a cycle doing nothing) · dashed = branch skipped entirely

When threads in a warp disagree about a branch, the hardware can't split them up — there's only one instruction pointer for all 32. So it runs both paths, masking off the lanes that didn't take each one. Masked lanes still consume the cycle. This is branch divergence, and it's the first thing every CUDA programmer learns to fear.

Watch the two extremes: if all 32 threads agree (slider at 0 or 32), the untaken path is skipped entirely and utilization stays at 100%. The worst case isn't 50/50 by much — any disagreement forces both paths.

Try: set the slider to 1. One disagreeing thread out of 32 costs the whole warp the entire else-path. Now check utilization at 0, 16, and 32.
Threads 0–15 take the IF, threads 16–31 take the ELSE. Divergent — or not?
Divergent, in this simulator — they're all in one warp. But in a real grid of thousands of threads, if branch direction aligns with warp boundaries (e.g. if (threadIdx.x / 32 % 2)), each warp is internally unanimous and there is no penalty at all. Same code, different data layout, 2× speed. GPU performance is full of this.

Chapter 1 · Lab 04 · Memory

The memory wall (and the trick that hides it)

Math is cheap. Moving data is expensive. A multiply takes a few cycles; fetching its inputs from VRAM takes hundreds. Click each level of the hierarchy to feel the distance.

round trip: —
Latencies are representative, not exact — real numbers vary by architecture and load. The shape never changes: each level is roughly an order of magnitude farther.

So a warp that touches VRAM just stalls for ~600 cycles? It would — and here is the single most important idea in GPU design: the SM doesn't wait, it switches. Each SM keeps dozens of warps resident (that's what the giant register file from Lab 02 is for — everyone's state stays loaded), and every cycle the warp scheduler simply picks any warp that isn't blocked. Memory latency isn't eliminated. It's hidden under other warps' work.

core utilization: —
Each row is a warp: ■ copper = computing · ▨ gray = waiting on memory. The top strip is the core itself — dark gaps are wasted cycles.
Try: find the smallest number of warps that gets the core to ~100%. That number — enough parallel work to cover your longest stall — is called occupancy, and chasing it is half of GPU optimization.
Why do GPUs get away with small caches when CPUs need huge ones?
A CPU cache exists to make one thread never wait. A GPU doesn't need any single thread to be fast — it just needs somebody to be ready. Latency tolerance via warp-switching substitutes for latency avoidance via caching, so the silicon goes to more cores instead.

Chapter 1 · Lab 05 · Graphics

Triangles in, pixels out

The "G" in GPU: everything you've ever seen on a screen was triangles. The rasterizer is fixed-function hardware that answers one question absurdly fast: which pixels does this triangle cover? Drag the vertices.

3 vertices → 0 fragments

Each cell is a pixel; the dot is its sample point. A pixel becomes a fragment if its sample point is inside all three edges — three sign checks, done by dedicated circuits for billions of pixels per second.

The colors blend because the hardware interpolates vertex attributes using barycentric coordinates — "how close am I to each corner?" The same machinery interpolates texture coordinates, depth, normals… and then a shader program (running on the SMs from Lab 02!) computes each fragment's final color.

This is where the GPU's shape came from. Vertices are independent → parallel. Fragments are independent → massively parallel. The 1990s built fixed pipelines for exactly this; the 2000s made the shading stages programmable; and by 2007 (CUDA) people realized the "fragments" don't have to be pixels at all. They can be anything.

Try: make a long, one-pixel-thin sliver of a triangle. Real GPUs shade pixels in 2×2 quads, so slivers waste up to 75% of the work — this is why game engines dread "micro-triangles".

Chapter 1 · Lab 06 · The bridge to AI

Tensor cores: multiply whole matrices per clock

Neural networks are, computationally, almost nothing but matrix multiplication. So in 2017 GPUs grew a new organ: the tensor core, a circuit that consumes an entire matrix tile in one operation instead of one number pair at a time. Race the two below.

B (4×4)

=

SCALAR CORE — one multiply-add per cycle

cycle 0 / 64

TENSOR CORE — the whole tile per cycle

cycle 0 / 1
Stylized: a real 4th-gen tensor core chews 16×16 tiles of FP16/FP8 numbers over a few cycles — the principle (amortize instruction overhead over a whole tile of fused multiply-adds) is exactly this.

Why does this work? A matrix multiply does O(n³) math on O(n²) data — the rare workload with enough arithmetic per byte to outrun the memory wall from Lab 04. Feed a tensor core once and it can grind for many cycles. That's why "AI FLOPS" numbers are so absurd: a single H100-class die does on the order of a quadrillion low-precision operations per second, nearly all of them inside these units.

Tensor cores also embrace low precision — FP16, FP8, even FP4 — because neural nets tolerate noisy arithmetic. Halve the bits, roughly double the throughput. Chapter 2 shows what all these matrix multiplies are actually for.

Your model is memory-bound (Lab 04 style). Do tensor cores help?
Barely. Tensor cores raise the math ceiling, not the memory ceiling. Small-batch LLM inference, for example, is dominated by streaming weights out of VRAM — the tensor cores sit mostly idle. This is why inference tricks (batching, quantization, KV-caches, speculative decoding) are mostly about bytes, not FLOPS.

Chapter 2 · Lab 08 · Transformers

Attention: how models decide what matters

Words mean nothing alone — "it" means nothing until you know what it points to. A transformer resolves this with attention: every token looks at every other token and decides how much to care. Hover any word to see where it looks. Then flip one word and watch the meaning of "it" physically move.

hovering: it
Weights are illustrative (hand-set to match how trained models resolve this classic Winograd sentence) — arc thickness = attention weight. In a real model this happens in every layer, in ~100 parallel “heads”, each looking for different relationships.

The trophy doesn't fit in the suitcase because it is too big — "it" is the trophy. Too small — "it" is the suitcase. No grammar rule decides this; physics does. The fact that attention learns to route this correctly, purely from predicting text, is the cleanest small demonstration of why transformers feel like they understand.

Mechanically each token asks a question (its query), every token advertises what it offers (its key), and the match scores — one dot product per pair, softmaxed — decide whose value gets blended in. All of it is matrix multiplies. All of it lands on the tensor cores from Lab 06. The circle closes.

Try: hover “because”, then “too”. Function words attend broadly and weakly; content words attend sharply. Then flip the sentence and hover “it” in both versions.

Simplified honestly: WQ, WK, WV are the identity here so you can trace every number; real models learn those matrices, which lets each layer ask different questions of the same vectors.

Read the matrix like this: row = who's asking, column = who's being looked at. "sat" scores high against "cat" and "mat" because their vectors share features, and softmax turns scores into a probability budget — attend more here, less there. The causal mask is what makes a model generative: during training, token 7 must not peek at token 8 (it's trying to predict it), so future scores are set to −∞ before the softmax. Toggle it and watch the matrix go triangular — you are looking at the single most consequential upper-triangle of zeros in modern computing.

Try: push sharpness β up and the softmax saturates — every token stares at one friend (and gradients die; this is why scores are divided by √d). Pull β down and attention smears into a uniform blur.
The matrix is 6×6 for 6 tokens. What happens at a 100k-token context?
Attention is O(n²) in sequence length: 100k tokens means 10 billion scores per head, per layer. This quadratic is why long context is expensive, why KV-caches exist (reuse the K and V you already computed), and why a zoo of sub-quadratic attention variants keeps being invented.

Chapter 3 · Lab 11 · Generation

One token at a time, forever

Everything a language model "writes" is this loop: read the context, produce a probability for every token in the vocabulary, pick one, append it, repeat. The picking rule is yours to control. This toy model's whole vocabulary is 16 tokens — play with the dials that shape every LLM you've ever used.

P(NEXT TOKEN | CONTEXT) — after temperature & top-k

distribution entropy: — bits
Temperature rescales the scores before softmax: cold → the favorite always wins (deterministic, repetitive); hot → the tail gets real probability (creative, then unhinged). Top-k simply deletes all but the k best before renormalizing.

Set temperature to 0.05 and sample twenty tokens: the model falls into its favorite rut and loops. Crank it to 3: grammar survives (the bigram structure) but sense dies. Real models live around 0.7–1.0, with top-p/top-k as guardrails. When an LLM "hallucinates confidently", remember what you're seeing here: sampling from a distribution never stops being sampling.

And each of those samples is one full forward pass — embeddings, every attention layer, every MLP — through the tensor cores of Lab 06. A thousand-word answer is a thousand trips through the whole machine, which is why generation speed is measured in tokens/second and why inference is memory-bound (Lab 06's checkpoint).

Try: temperature 0.05, sample ×10, RESET, repeat — same rut every time? Now k=3 at temperature 2 — chaos, but bounded chaos. That combination (hot but clipped) is basically "creative writing mode".

Chapter 4 · Lab 12 · Calculus

The chain rule is the whole trick

All of deep learning rests on one idea from calculus: if you know how each small step transforms a number, you can run the chain backwards and ask "how much was each input to blame for the final error?" Here is one neuron — z = w·x + b, squashed by tanh, scored against a target. Every gradient below is live.

loss — · steps 0
Copper numbers under each edge are gradients: ∂loss/∂(that thing). Each is just (local derivative) × (gradient flowing in from the right) — multiplication, chained. That's backpropagation. All of it.

Follow one: ∂L/∂w = ∂L/∂a · ∂a/∂z · ∂z/∂w = 2(a−y) · (1−a²) · x. Three local facts multiplied. A 100-billion-parameter model is this exact bookkeeping over a graph with billions of nodes — computed in one backward sweep that costs about as much as two forward passes.

Now click TAKE A GRADIENT STEP a few times and watch the loss fall: you're running Lab 07's training loop by hand. Notice tanh's danger: push w and x so that a saturates near ±1, and (1−a²)→0 — the gradient vanishes and learning stops. That single failure mode shaped a decade of architecture design (ReLU, residual connections, normalization).

Try: set x=3, w=3 — a pins to 1.0 and every gradient dies. Then find a setting where one step of gradient descent overshoots and the loss goes up. You've just reproduced Lab 13's learning-rate problem, which is next.

Chapter 4 · Lab 13 · Optimization

Rolling downhill in the dark

Training = finding low ground on a surface you can't see, using only the local slope. Click anywhere to drop a ball, then tune the two dials every ML engineer tunes: learning rate and momentum. Each preset is a pathology you will meet in real training runs.

loss — · steps 0 ·
Darker = lower. The ball follows −∇f, scaled by the learning rate; momentum blends in the previous step's direction.

The RAVINE is the one to internalize — a narrow curved valley (this is Rosenbrock's function, optimization's favorite villain). Plain gradient descent ping-pongs across the steep walls while crawling along the floor. Add momentum ~0.9 and the sideways oscillations cancel while the along-the-valley component accumulates: this is why Adam and friends exist — real loss surfaces in billions of dimensions are ravines all the way down.

On TWO MINIMA, where you drop the ball decides which basin you end in — same landscape, different solution. That's Lab 07's RE-INITIALIZE button, seen from above. And the SADDLE: flat enough to stall a ball with no momentum, and in high dimensions saddles vastly outnumber true minima.

Try: BOWL with learning rate maxed — watch it diverge (the flash is real; this is the NaN you'll someday see in a training log). Then beat the RAVINE: fastest ball to the minimum wins; you'll need both dials.
Why not just use a huge learning rate with huge momentum everywhere?
The step size must respect the surface's curvature: too big and each step overshoots the valley floor, gaining height instead of losing it — momentum then amplifies the explosion. Adaptive optimizers (Adam, RMSProp) estimate per-parameter curvature on the fly, which is why they're the default; warmup and LR schedules exist because the safe step size changes as training moves through different regions.

Chapter 4 · Lab 14 · Linear algebra

A matrix is a warp of space

Stop reading matrices as number grids. A 2×2 matrix is fully described by where it sends two arrows: drag the copper and blue handles — they are the columns of the matrix — and watch all of space follow.

M = [1 0; 0 1] det = 1.00
Shaded square = image of the unit square; |det| is its area (negative = space flipped). Dashed lines are eigenvector directions — the axes the warp only stretches, never turns.

Now the payoff for chapters 1–3: a neural network layer is exactly this warp — in a few thousand dimensions — followed by a shift (the bias) and a fold (the nonlinearity). Warp, shift, fold, repeat twelve or a hundred times: that's GPT. Lab 07's decision boundary was built from these moves; you watched space being kneaded until copper and blue became linearly separable.

Hit COLLAPSE: determinant zero, 2-D space flattened onto a line — information destroyed, unrecoverable. Real layers must avoid becoming accidentally collapse-like (rank collapse), and eigen-thinking is everywhere: the stability of training, PCA, why deep networks explode or vanish (eigenvalues of repeated warps compound exponentially), even Google's original PageRank.

Try: make a pure rotation — the eigenvector lines vanish (no direction survives unturned; the eigenvalues went complex). Then drag both handles onto the same line and watch det hit 0 at the exact moment the plane dies.

Chapter 5 · Lab 16 · Memory patterns

Feeding the beast: coalescing

Lab 04 taught that bytes, not FLOPS, are the budget. Here's the knife-edge version every CUDA programmer meets in week one: the same 32 loads cost 1 memory transaction or 32, depending only on the pattern. Drag the stride.

float v = a[tid * 1];   // one warp, 32 threads
transactions: 1 · bandwidth efficiency: 100%
Memory arrives in 128-byte lines (32 floats). Stride 1: the warp's 32 loads land in one line — one transaction. Stride 32: every thread pulls its own line and throws 97% of each away.

This is why "array of structs vs. struct of arrays" is a religious war GPU people always win the same way: neighboring threads must touch neighboring bytes. And it's the first thing to check when a kernel underperforms — on your Orin's ~68 GB/s (shared with the CPU!), wasted bandwidth hurts even more than on a datacenter card.

The second classic trick is tiling: when data is reused (matmul reuses every element n times), stage it in the SM's shared memory (Lab 02) once, then hammer it from there. Naive matmul reads each element from VRAM n times; a tiled kernel reads it n/TILE times. This one idea is most of the distance between a slow matmul and cuBLAS — and writing the tiled version yourself is the single best CUDA exercise that exists. It's week 2 of your curriculum.

Try: find the stride where efficiency halves (2), then the cliff's bottom (32). Then, on the Jetson: benchmark a[i] vs a[i*32 % N] over 100M elements and watch the theory show up in milliseconds.
A kernel reads a matrix row-by-row per thread and runs 10× slower than expected. First hypothesis?
Row-per-thread means consecutive threads start 1 row apart — stride = row width, the worst pattern on this slider. Either transpose the access (thread-per-column so neighbors read neighbors), or tile through shared memory. Nsight Compute will show it instantly as low "global load efficiency" — learn to read that number early.

Chapter 6 · Lab 17 · Invariants

Halve the world until it confesses

Binary search is not "check the middle." It is a promise you refuse to break: the answer is inside [lo, hi]. Every comparison is chosen so that, whatever it says, you can throw away half the window and the promise still holds. Watch it race a linear scan on the same array — same target, same data, one of them provably wasteful.

linear 0 · binary 0 · window 64
worst case — linear 64 · binary 6 steps
Top row: the linear scan, paying one comparison per element. Bottom row: the dimmed bars are dead — provably outside [lo, hi], never to be touched again. The copper bar is the next question. Drag the slider to N = 4 billion: linear needs 4 billion steps, binary needs 32.

Two things make this an interview weapon. First, the invariant discipline: every off-by-one bug in every binary search ever written is a broken promise — a mid that leaks back into the window, a hi that excludes the answer. Decide what [lo, hi] means before you type, and the loop writes itself.

Second: flip PREDICATE MODE. The bars stop being values and become answers to a yes/no question — F F F F T T T T. Any question that flips once and never flips back is binary-searchable, even when there's no array anywhere: the first build that fails, the minimum speed that finishes in time, the largest tile that fits in memory. Rephrase "optimize x" as "is x enough?" and an impossible problem becomes 32 questions. That reframe — binary search on the answer — is a full third of the hard problems on your ladder.

# first index where pred(i) is True — the only form you need
def first_true(lo, hi, pred):          # answer in [lo, hi]
    while lo < hi:
        mid = (lo + hi) // 2
        if pred(mid):
            hi = mid       # mid might BE the answer — keep it
        else:
            lo = mid + 1  # mid is proven false — discard it
    return lo                          # lo == hi: the promise, kept

# find x in sorted a:  first_true(0, len(a), lambda i: a[i] >= x)
// same invariant, same two lines that matter
int first_true(int lo, int hi, auto&& pred) {   // answer in [lo, hi]
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;  // no overflow, unlike (lo+hi)/2
        if (pred(mid)) hi = mid;       // mid might be the answer
        else           lo = mid + 1;  // mid is proven false
    }
    return lo;
}
// the STL already knows this form:
//   std::lower_bound(a.begin(), a.end(), x)  — first element >= x
//   std::partition_point(f, l, pred)         — first_true, verbatim
Try: STEP through a binary run and, before each step, say out loud which half dies and why the promise survives. Then set the slider to 32: four billion elements, thirty-two questions. That ratio — not the code — is what the interviewer is checking you feel.

Chapter 6 · Lab 18 · Graph search

One loop, three buckets

DFS, BFS, and Dijkstra are marketed as three algorithms. They are one loop: pull a cell from the frontier, visit its neighbors, repeat. The only thing that changes is the bucket holding the frontier — a stack, a queue, or a min-heap — and that single swap changes everything you see below.

explored 0 · path cost —
Blue wash = visited, brighter = earlier. Copper rim = the frontier. Murky green = swamp (costs 5× to cross). Watch BFS on SWAMP march proudly straight through the expensive mud — it counts steps, not cost. Dijkstra, same loop with a heap, walks around it.

The shapes are the lesson. DFS is a spelunker: it commits to one corridor to the bitter end, and the path it returns is whatever wandering route it happened to survive — found ≡ shortest is false. BFS is a ripple: it exhausts distance k before touching k+1, so the first time it reaches the goal is provably the fewest-steps route. Dijkstra is the same ripple in cost-space instead of step-space. (Give the heap a hunch about direction and you've invented A* — that's the whole extension.)

The interview skill is rarely writing the loop — it's seeing the graph where none was drawn. Word ladders, lock combinations, course prerequisites, "minimum moves to…": states are nodes, legal moves are edges, and the question "fewest moves?" is the word BFS wearing a costume. Say what the bucket is and why, and the rest is typing.

import heapq
def dijkstra(adj, s):                  # adj[u] = [(v, w), ...]
    dist = {s: 0}
    pq = [(0, s)]                      # the bucket. that's the algorithm.
    while pq:
        d, u = heapq.heappop(pq)
        if d > dist[u]: continue       # stale entry — skip
        for v, w in adj[u]:
            if d + w < dist.get(v, 1e18):
                dist[v] = d + w
                heapq.heappush(pq, (dist[v], v))
    return dist
# BFS: pq → collections.deque, popleft(), all w == 1
# DFS: pq → plain list, pop()  — and the guarantees evaporate
#include <queue>
#include <vector>
using P = std::pair<long long, int>;             // (dist, node)
std::vector<long long> dijkstra(const auto& adj, int s) {
    std::vector<long long> dist(adj.size(), LLONG_MAX);
    std::priority_queue<P, std::vector<P>, std::greater<>> pq; // the bucket
    dist[s] = 0; pq.push({0, s});
    while (!pq.empty()) {
        auto [d, u] = pq.top(); pq.pop();
        if (d > dist[u]) continue;               // stale entry — skip
        for (auto [v, w] : adj[u])
            if (d + w < dist[v]) { dist[v] = d + w; pq.push({dist[v], v}); }
    }
    return dist;
}
// BFS: pq → std::queue, all w == 1 · DFS: → std::stack (or recursion)
Try: run all three on SWAMP and read the two numbers each leaves behind — cells explored vs. path cost. DFS explores least here and pays most; Dijkstra explores most and pays least. That trade is the entire theory of search, in two integers.

Chapter 6 · Lab 19 · Dynamic programming

Remembering is the whole algorithm

DP has a fearsome reputation and a one-sentence definition: recursion that refuses to redo work. Below is naive fib(n) as an actual call tree — every node one function call, every color one argument. Same color = same computation, repeated. Then flip the switch that remembers.

calls: naive — · memoized —
Every same-colored subtree is the same answer being re-derived from scratch. MEMOIZE keeps only the first of each color; the hollow dots are cache hits — a lookup, not a subtree. The tree of 177 calls becomes 19.
work 0 / 70 cells · greedy — vs optimal —
Cheapest top-left → bottom-right path, moving only right or down. Each cell needs one fact: cost + min(from-above, from-left). 70 cells of work stands in for the 5,005 complete paths a brute force would walk. GREEDY always grabs the locally cheaper neighbor — watch it lose to the table.

Every DP interview reduces to two questions, asked in this order. One: what is the state? — the arguments of your recursion, the minimum description of "where am I" (a fib argument; a grid cell; (index, remaining capacity) for knapsack). Two: where does the answer sit? — one corner of the table, or a max over the last row. Answer those and the recurrence is one honest line; filling order and loops are bookkeeping. When you're stuck, you're stuck on question one — naming the state — not on "DP".

The two stages above are the same idea from both ends: memoization is the recursion writing the table lazily on demand; tabulation is you filling it in a deliberate order. Interviews accept either — say the state out loud, write @cache or the loops, and mention that the tree you didn't build was exponential. Ladder tie-in: 1-D states week 6, 2-D states week 8.

from functools import cache

@cache                                 # memoization: the tree, pruned
def fib(n):
    return n if n < 2 else fib(n-1) + fib(n-2)

def min_path(g):                       # tabulation: the table, filled
    R, C = len(g), len(g[0])
    dp = [[0]*C for _ in range(R)]
    for r in range(R):
        for c in range(C):
            best = min(dp[r-1][c] if r else 1e18,
                       dp[r][c-1] if c else 1e18)
            dp[r][c] = g[r][c] + (0 if r == c == 0 else best)
    return dp[-1][-1]                  # the answer sits in the corner
#include <vector>
long long fib(int n, std::vector<long long>& m) {   // memoization
    if (n < 2) return n;
    if (m[n] != -1) return m[n];                    // cache hit — no subtree
    return m[n] = fib(n-1, m) + fib(n-2, m);
}

long long min_path(const std::vector<std::vector<int>>& g) {
    size_t R = g.size(), C = g[0].size();           // tabulation
    std::vector<std::vector<long long>> dp(R, std::vector<long long>(C));
    for (size_t r = 0; r < R; ++r)
        for (size_t c = 0; c < C; ++c) {
            long long up   = r ? dp[r-1][c] : LLONG_MAX;
            long long left = c ? dp[r][c-1] : LLONG_MAX;
            dp[r][c] = g[r][c] + ((r||c) ? std::min(up,left) : 0);
        }
    return dp[R-1][C-1];
}

One switch, exponential → linear. That gap is why "have we computed this before?" is the most valuable question in the interview.

Chapter 6 · Lab 20 · Hashing

The oldest trade: memory for time

Half the easy problems on your ladder are one question in a costume: “have I seen this before?” An array answers it in O(n) by re-reading its whole life. A hash table answers in O(1) by spending memory: scramble the key into a bucket number, and the answer is waiting in that bucket. Below, watch the buckets fill — and watch what happens when the scrambling is lazy.

n 0 · load α 0.00 · longest chain 0
Each column is a bucket; each block a key. A lookup walks one column top-down — so the longest chain is your real worst case, and the load factor α = n/buckets is what you manage. Past α ≈ 1, RESIZE ×2 and re-scatter. Toggle BAD HASH: same keys, same table, and suddenly four columns do all the work — O(1) was never a property of the table, only of the scatter.

The contract has two clauses, and both are yours to keep: the hash must spread (real libraries mix every bit of the key; BAD HASH above keeps only the low bits — watch the clustering), and the table must grow before chains do (resize at a load threshold; amortized, still O(1)). Break either clause and every “O(1)” in your complexity analysis quietly becomes O(n) — which is precisely the follow-up question interviewers like to ask.

On the ladder (week 1), the table appears as three reflexes: the seen-set (duplicates, cycles), the complement lookup (two-sum: for each x, is target−x already in the table?), and the frequency map (anagrams, counts, group-by). All three are the same move — pay one insert now so every future question costs O(1). And know the one thing the table can't do: it has no notion of order. The moment the question says “nearest,” “smallest above,” or “in sorted order,” you want a sorted structure or a heap (Lab 22), not a dict.

# the three reflexes, week 1 of the ladder

std::pair<int,int> two_sum(const std::vector<int>& xs, int t) {
    std::unordered_map<int,int> at;     // value -> index
    for (int i = 0; i < (int)xs.size(); ++i) {

            return {it->second, i};
        at[xs[i]] = i;
    }

}

Chapter 6 · Lab 21 · Two pointers

Two pointers, zero regrets

A brute force re-examines; two pointers never look back. The license for that confidence is always some monotone fact — “once a window is invalid, growing it can't fix it” — that lets you prove entire regions dead without visiting them. Below: the longest run of unrepeated characters, and a converging two-sum. Both pointers only ever move forward; count the steps.

pointer moves 0
WINDOW: copper bracket = L, blue bracket = R. R eats forward; when it swallows a repeat, L leaps past the old copy — never rewinding. Every cell L abandons is proven useless: any window containing the repeat is already invalid. Total pointer moves ≤ 2n, against the 210 windows a brute force would inspect.

The two shapes cover week 2 of the ladder between them. The window (L chases R): substrings and subarrays with a constraint — longest without repeats, shortest with sum ≥ target, at most k distinct. The invariant is “the window is always valid”; R expands until it breaks, L contracts until it's repaired, and each index enters and leaves at most once. The converge (L and R close in): pair-hunting in sorted data — two-sum, container with most water, trapping rain. There the monotone fact is starker: if the sum is too small, nothing paired with the current L can ever work, so L's entire column of possibilities dies in one comparison. It's Lab 17's halving argument wearing running shoes.

Know when the license is revoked: “shortest subarray with sum ≥ k” with negative numbers breaks the window's monotonicity (growing can now fix an invalid window), and the honest answers are prefix sums with a monotonic deque, or a different tool entirely. Saying that unprompted — “two pointers needs monotonicity, negatives break it” — is the difference between pattern-matching and understanding, and interviewers are listening for exactly it.

def longest_unrepeated(s):              # the window skeleton
    last, L, best = {}, 0, 0
    for R, ch in enumerate(s):
        if ch in last and last[ch] >= L:
            L = last[ch] + 1            # leap — never rewind
        last[ch] = R
        best = max(best, R - L + 1)     # window is valid here
    return best

def two_sum_sorted(a, t):               # the converge skeleton
    L, R = 0, len(a) - 1
    while L < R:
        s = a[L] + a[R]
        if s == t: return L, R
        if s < t: L += 1                # all pairs (L, *) proven dead
        else:     R -= 1
    return None
#include <array>
int longest_unrepeated(const std::string& s) {
    std::array<int,256> last; last.fill(-1);
    int L = 0, best = 0;
    for (int R = 0; R < (int)s.size(); ++R) {
        if (last[(unsigned char)s[R]] >= L)
            L = last[(unsigned char)s[R]] + 1;  // leap — never rewind
        last[(unsigned char)s[R]] = R;
        best = std::max(best, R - L + 1);
    }
    return best;
}
std::pair<int,int> two_sum_sorted(const std::vector<int>& a, int t) {
    int L = 0, R = (int)a.size() - 1;
    while (L < R) {
        int s = a[L] + a[R];
        if (s == t) return {L, R};
        if (s < t) ++L; else --R;       // a whole row of pairs dies
    }
    return {-1,-1};
}

Then run CONVERGE and count moves: n=16 elements, ≤15 comparisons, versus the 120 pairs brute force would check. The ratio is small here — at n=10⁵ it's the difference between an accepted OA and a timeout.

Chapter 6 · Lab 22 · Heaps

An array wearing a tree costume

A heap makes exactly one promise — the smallest thing is on top — and keeps it lazily: parents beat children, siblings are strangers. That laziness is the speed. And the tree is a costume: the whole thing lives in a flat array where node i's children sit at 2i+1 and 2i+2, no pointers anywhere. Push a value and watch both views move as one.

size 7 · last op — swaps · bound ⌈log₂n⌉ = 3
Same fifteen slots, two costumes: the tree you reason in, the array it actually is. Cooler = smaller. A PUSH lands in the next free slot and sifts up its ancestor ladder; POP MIN crowns the last leaf and sifts it down. Every swap flashes in both views at once — there is no second data structure, only a second way of looking.

The costume trick is why heaps are everywhere: an insert touches only the ancestor ladder — ⌈log₂n⌉ slots, 20 for a million items — and the array layout means those touches are cache-friendly integer arithmetic, not pointer chasing. HEAPIFY is the quieter surprise: building from scratch bottom-up costs O(n), not n log n, because almost all nodes are leaves and sift barely anywhere. Interviewers spring that one as a follow-up; now you've watched it happen.

Week 5 of the ladder is mostly one pattern: top-k. Keep a heap of size k of the best-so-far, and its top is the worst of the best — the gatekeeper each new candidate must beat. n log k, tiny memory, works on streams you can't hold. (k large relative to n? Quickselect, next lab, does it in linear time.) The other week-5 shape, intervals, is heaps too: sweep sorted starts, park end-times in a min-heap, and “rooms needed” is just the heap's high-water mark. And you've already run the loop that matters most: Dijkstra in Lab 18 — its bucket is exactly this structure.

import heapq                            # min-heap. always. no flag.
h = [5, 9, 12]
heapq.heapify(h)                        # O(n), in place
heapq.heappush(h, 3)                    # sift-up,  O(log n)
smallest = heapq.heappop(h)             # sift-down, O(log n)

def top_k(xs, k):                       # the week-5 pattern
    h = []
    for x in xs:
        if len(h) < k: heapq.heappush(h, x)
        elif x > h[0]: heapq.heapreplace(h, x)  # beat the gatekeeper
    return sorted(h, reverse=True)      # n log k, stream-safe

# max-heap in python: push -x, negate on the way out. yes, really.
#include <queue>

std::priority_queue<int> maxh;                       // top() = largest
std::priority_queue<int, std::vector<int>,
                    std::greater<>> minh;            // top() = smallest

std::vector<int> top_k(const std::vector<int>& xs, size_t k) {
    std::priority_queue<int, std::vector<int>, std::greater<>> h;
    for (int x : xs) {
        if (h.size() < k) h.push(x);
        else if (x > h.top()) { h.pop(); h.push(x); } // beat the gatekeeper
    }
    std::vector<int> out;
    while (!h.empty()) { out.push_back(h.top()); h.pop(); }
    return {out.rbegin(), out.rend()};
}

Chapter 6 · Lab 23 · Sorting & selection

Sort half as much as you think

One move powers this whole lab: partition — pick a pivot, sweep once, and the array splits into smaller-than and larger-than, pivot seated in its final home forever. Recurse on both sides and you invented quicksort. Recurse on one side — because the k-th element can only be in one of them — and you get quickselect: the k-th smallest of anything, in linear expected time, no sorting.

comparisons 0 · n log₂n ≈ 268 · n²/2 = 1152
Copper bar = current pivot; the bright band = the active partition; blue bars are seated forever. QUICKSELECT is the reveal: after each partition, the half that provably can't hold the k-th goes dark and stays dark — Lab 17's halving argument, except the halves are earned with a sweep instead of given by sorted order. WORST CASE feeds quicksort sorted input with an end pivot: watch the comparison counter walk toward n²/2.

Three honest facts about sorting in interviews. One: you will call sort(), not write it — what's tested is knowing the n log n price, and spotting when sorting first turns a hard problem into an easy one (intervals, ladder week 5, is exactly this). Two: the comparison bound is real — n log n is the floor for comparison sorting, so any claimed faster sort is exploiting structure (counting sort on small ints, radix on fixed-width keys); saying which structure is the whole answer. Three: WORST CASE isn't a party trick — sorted input with a naive pivot is quicksort's n², which is why real libraries randomize or use introsort, and why "how would an adversary hurt this?" is a fair question about any algorithm you name.

Quickselect is the one to actually carry: “k-th largest,” “median,” “top half” without n log n. The decision rule for week 5-and-beyond: k small or the data streams → heap (Lab 22, n log k); k anywhere and the data sits in memory → quickselect (expected O(n)); need the k results in order → sort and stop apologizing. And name-drop with precision: std::nth_element is quickselect, one call, already in the standard library — knowing that is worth more than reciting the partition loop.

import random
def quickselect(a, k):                  # k-th smallest, 0-indexed; avg O(n)
    lo, hi = 0, len(a) - 1
    while True:
        p = partition(a, lo, hi, random.randint(lo, hi))
        if p == k: return a[p]
        if p < k: lo = p + 1            # left half can't hold the k-th
        else:     hi = p - 1            # right half can't either

def partition(a, lo, hi, pi):           # lomuto: one sweep, pivot seated
    a[pi], a[hi] = a[hi], a[pi]
    store = lo
    for i in range(lo, hi):
        if a[i] < a[hi]:
            a[store], a[i] = a[i], a[store]; store += 1
    a[store], a[hi] = a[hi], a[store]
    return store                        # final home, forever
#include <algorithm>
// the library already contains this lab:
std::vector<int> v = data;
std::nth_element(v.begin(), v.begin() + k, v.end());
// v[k] is now the k-th smallest; left of it smaller, right larger,
// neither side sorted — expected O(n), and saying so is the answer.

int partition(std::vector<int>& a, int lo, int hi) {  // lomuto, end pivot
    int store = lo;
    for (int i = lo; i < hi; ++i)
        if (a[i] < a[hi]) std::swap(a[store++], a[i]);
    std::swap(a[store], a[hi]);
    return store;                       // pivot seated, forever
}

// and bails to heapsort before an adversary can make it quadratic.

Chapter 6 · Lab 24 · Union-find

“Same tribe?” in constant time

Connectivity questions don't need the graph — only a chain of who answers for whom, flattened a little more every time you ask. Below: Kruskal's MST, which is nothing but “sort the edges, then ask same tribe? n times.”

asked 0 · kept 0/23 · refused 0
Edges arrive cheapest-first. Same color = same tribe: those edges are refused (faint dashes) — they'd close a cycle. Different tribes merge on contact. 23 acceptances later: the minimum spanning tree, guaranteed.

The insight to carry: when a problem merges groups over time — islands appearing, accounts sharing emails, network cables landing — the answer is a parent array, not a graph traversal. find walks to the root and flattens the path behind it; union points one root at the other. Amortized cost: effectively constant (inverse Ackermann — say “≤4 for any n that fits in the universe”). Kruskal's greedy is legal by the cut property: the cheapest edge crossing any divide is always safe — your first exchange argument, one lab early.

class DSU:
    def __init__(self, n): self.p = list(range(n))
    def find(self, x):
        while self.p[x] != x:
            self.p[x] = self.p[self.p[x]]   # flatten as you climb
            x = self.p[x]
        return x
    def union(self, a, b):
        ra, rb = self.find(a), self.find(b)
        if ra == rb: return False           # same tribe — cycle
        self.p[ra] = rb; return True

# kruskal: for w,u,v in sorted(edges): keep edge if dsu.union(u,v)
struct DSU {
    std::vector<int> p;
    DSU(int n) : p(n) { std::iota(p.begin(), p.end(), 0); }
    int find(int x) {
        while (p[x] != x) { p[x] = p[p[x]]; x = p[x]; }  // flatten
        return x;
    }
    bool unite(int a, int b) {
        a = find(a); b = find(b);
        if (a == b) return false;           // same tribe — cycle
        p[a] = b; return true;
    }
};
// kruskal: sort edges by weight, keep those where unite() says true

Chapter 6 · Lab 25 · Greedy

Greed you can prove

Three plausible strategies, one interview classic: schedule the most non-overlapping intervals. Two of these lose badly — greedy is not a vibe, it's a swap-proof, and only one of them has it.

kept — start ? · short ? · finish ? (max possible: 7)
Copper = kept, dark = refused, outline = being considered. Same ten intervals every run — only the order of asking changes.

Why earliest-finish wins, in one move: take any optimal schedule and swap its first interval for the earliest-finisher — it ends no later, so everything after still fits. Repeat down the line: optimal morphs into greedy, losing nothing. That's an exchange argument, and it's the entire license. The interview discipline: if you can't sketch the swap in two sentences, don't trust the greedy — add weights to these same intervals and earliest-finish breaks, and the answer is DP (Lab 19). Greedy-with-a-license runs this whole chapter: Dijkstra, Kruskal, Huffman.

def max_meetings(iv):
    iv.sort(key=lambda x: x[1])          # by END. that's the theorem.
    done, t = 0, float('-inf')
    for s, e in iv:
        if s >= t: done, t = done + 1, e
    return done

Chapter 6 · Lab 26 · Topological order

Peel the free nodes

Anything with prerequisites — courses, builds, your own summer — is a DAG, and ordering it is one move repeated: peel everything that depends on nothing. If the peel ever stalls, you haven't failed — you've proven a cycle exists. Detection comes free.

waves 0 · ordered 0/10
Your actual pipeline. Each wave (numbered) is everything currently free — peel it, arrows die, new nodes free up. The toggle adds OFFER → JETSON: “I'll build once I have the offer.” Run the peel and watch the whole plan deadlock — nothing is free, ever. That stall is the cycle proof.

Interview reflex: the phrases “prerequisites,” “build order,” “must come before” all mean Kahn's algorithm — count in-degrees, queue the zeros, peel. Course Schedule I is “does the peel finish?”, II is “print the peel order,” and alien dictionary is the same peel after you extract the edges. One upgrade worth knowing: process the DAG in peel order and you can DP along it — longest path in a DAG is exactly that, and “longest chain of prerequisites” questions are it in costume (Lab 19 meets this lab).

from collections import deque
def topo(n, adj):
    indeg = [0]*n
    for u in range(n):
        for v in adj[u]: indeg[v] += 1
    q = deque(i for i in range(n) if indeg[i] == 0)   # the free nodes
    order = []
    while q:
        u = q.popleft(); order.append(u)
        for v in adj[u]:
            indeg[v] -= 1
            if indeg[v] == 0: q.append(v)             # newly freed
    return order if len(order) == n else None        # None ⇒ cycle. free.
std::vector<int> topo(int n, const std::vector<std::vector<int>>& adj) {
    std::vector<int> indeg(n, 0), order;
    for (int u = 0; u < n; ++u) for (int v : adj[u]) ++indeg[v];
    std::queue<int> q;
    for (int i = 0; i < n; ++i) if (!indeg[i]) q.push(i);  // free nodes
    while (!q.empty()) {
        int u = q.front(); q.pop(); order.push_back(u);
        for (int v : adj[u]) if (--indeg[v] == 0) q.push(v);
    }
    if ((int)order.size() != n) return {};   // empty ⇒ cycle. free.
    return order;
}
